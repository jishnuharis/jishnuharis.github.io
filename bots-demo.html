How it works

Users send /find to join a queue. The bot pairs two strangers instantly and relays every message bidirectionally and neither side ever sees the other's Telegram ID. Admin controls for moderation are baked in with an integrated rating system to help filter out abuse of the bot.

The idea

I wanted to build something that didn't feel like a toy bot. Most "random chat" bots online are barebones, no persistence, no real safety net. I wanted one that actually held state properly: profiles, ratings, reports, even a small game to give people a reason to stick around in a chat instead of just skipping immediately. The whole thing runs without storing any third-party data or accounts, just Telegram IDs, paired and forgotten once the chat ends.

Architecture

The bot is built on python-telegram-bot v20+ with async handlers and a job-queue for periodic tasks. Two users land in a shared waiting queue via /find, and the moment there are two in the queue, they're paired and the bot starts forwarding everything between them (text, photos, video notes, voice messages, stickers, GIFs) without either side ever seeing the other's real Telegram ID. User data (profile, points, ratings) lives in the database, but instead of writing on every single change, I track which users are "dirty" in a set and flush only those every 60 seconds whixh cuts down on database load a lot compared to writing the whole table constantly. A Flask keep-alive server runs alongside it so cloud platforms don't spin the bot down for being idle.

Challenges

The trickiest part was state recovery: if the bot restarts mid-conversation (deploys happen), I needed every active pairing restored from the database on boot, not just user profiles, otherwise people get silently disconnected and have no idea why. Double-voting was another one: nothing stopped a user from spamming 👍/👎 on the same partner over and over which can be misused if left unchecked, so I built a feedback track system to block repeat votes within a session, then clear it out every 8 hours so it doesn't bloat memory forever. I also had to be careful with Telegram's API itself, users block the bot constantly, so every API call routes through a wrapper that swallows Forbidden errors instead of crashing the whole update loop. And for real bugs, instead of just logging to a file I never check, I built a global error handler that DMs me the full traceback (exception type and exact file/line) straight to my own Telegram on every unhandled exception.

What I'd improve

Right now matching is fully random from the queue and the time people joined the queue, even though I collect gender, age, and country during setup, adding actual filtering on those would make matches more relevant instead of just throwing two random people together. I'd also like rate-limiting on /next itself, since right now nothing stops someone from skipping partners as fast as they can type. Longer term, I want to expand the Coin Steal game into something with more rounds and stakes. Right now it's a fun side feature, but there's room to make it a bigger part of why people stay in a chat.
