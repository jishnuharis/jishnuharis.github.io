StormDrive

A single-user private archive bot living entirely inside Telegram, built for one person to store, browse, and organize files with zero hosting fees. Almost everything is driven by tappable inline-keyboard menus rather than typed commands, with atomic writes, 5 rolling backups, and a guarded archive group that the bot auto-kicks any intruder by itself.

What it does

A private archive vault for exactly one person, every interaction checks against an owner ID and silently refuses anyone else. Browsing, renaming, moving, deleting, and favouriting all happen by tapping inline-keyboard buttons, not typing paths.

The idea

I wanted a private place to dump files without paying for cloud storage or trusting a third-party service with my stuff. Most self-hosted solutions are overkill, servers, databases, web UIs, when all I really needed was something I already had open all day: Telegram. So I built a bot that lives entirely inside a single chat, where I can store, browse, and retrieve anything through tappable menus instead of typing paths or opening a browser.

Architecture

The bot is built on python-telegram-bot v20+ with a fully async handler setup. Files aren't stored on any server, they go straight into a private Telegram archive group, and the bot keeps a local JSON metadata database that maps every file to its Telegram file ID, folder path, type, and size. All writes are atomic with 5 rolling backups, so a corrupt write never destroys the whole index. The archive group itself is guarded, the bot auto-kicks anyone who isn't the owner the moment they join and notifies the owner about the intruder, so there's no risk of accidentally exposing files by sharing a link. The /disk generates a report of the current state of the bot will all the file details globally and sends it as an image to make it more readable. A Flask keep-alive server runs alongside it so the bot doesn't get spun down on when it's on idle mode.

Challenges

The trickiest part was making the whole thing feel like a real file manager without a real UI. Every screen, browsing folders, viewing file options, moving files, had to be built entirely from Telegram's inline keyboard buttons, with the bot editing its own previous message instead of flooding the chat with new ones. Getting that edit-in-place behaviour right took careful state threading: every button's onClick closure needs to carry the right folder path and file ID forward, and Telegram has strict rate limits on how fast you can call editMessageReplyMarkup. Atomic writes were another one, JSON isn't a database, so I had to write to a temp file and rename it over the original to make sure a crash mid-write never leaves a half-written index behind.

What I'd improve

Right now the metadata database is a flat JSON file, which is fine for hundreds of files but would slow down noticeably at tens of thousands. Switching to SQLite would fix that and also make search much faster than the current linear scan. I'd also like to add tag support, right now organisation is purely folder-based, but being able to tag files and filter by tag would make retrieval a lot more flexible. Longer term, I want to add a /share command that generates a time-limited link to a single file without exposing the archive group itself.
